Frame V2 API

Aircash Frame integration enables multiple Aircash services using a single endpoint. Connect your cashier directly to the Aircash Frame API to process:

  1. Payments via Aircash application
  2. Deposits via Abon voucher
  3. Withdrawals to Aircash application

SDK

General

There are two ways to integrate with Aircash Frame; Window Checkout and Redirect Checkout. Each of them will be described in its own section, including the required parameters and how to use the script after initiating. To use the Aircash Frame checkout script, the partner’s system must include a minified Javascript file on the web application. The partner's system is also responsible to ensure that script is loaded before calling any of the functions in it. This is an example how to load/include script: 

< script src="https://firebasestorage.googleapis.com/v0/b/aircash-dev/o/Frame%2Fjs%2Faircashframe-sdk-web.min.js?alt=media">< /script>

Aircash will provide a link to the script additionally.

Integration types

Window Checkout

This integration type offers you a solution where, with a few passed parameters to Aircash Frame SDK, script opens a new window with Aircash Frame in it. After finishing the transaction it will close and handle the rest of the process for you. Below are the required parameters for initiating a transaction: 

  {
      ... //rest of the parameters for API v2
      "successUrl": null,                                     //this must be 'null' when using this type of checkout
      "declineUrl": null,                                     //this must be 'null' when using this type of checkout
      "cancelUrl": null,                                      //this must be 'null' when using this type of checkout
      "originUrl": "https://domain.com",                      //this must be domain where your web application runs Aircash Frame SDK
      ... //rest of the parameters for API v2
  }

After a successful transaction initiate Aircash Frame will return the following response: 

  {
      "transactionId": "c125ba7d-afb5-4ce5-ac3c-33af420200cf",
      "url": "https://frame.aircash.eu/c125ba7d-afb5-4ce5-ac3c-33af420200cf"
  }

When the partner's system gets a response from Aircash Frame it's necessary to call the SDK function which can be done as in the following example: 

  new AircashFrame.WindowCheckout({
      transactionId: response.transactionId,                  //this is a "TransactionId" parameter from the Aircash Frame system response
      onSuccess: onSuccess,                                   //a function that's needed to be defined in the partner's system, it's called after successful transaction
      onDecline: onDecline,                                   //a function that's needed to be defined in the partner's system, it's called after declined transaction
      onCancel: onCancel,                                     //a function that's needed to be defined in the partner's system, it's called after transaction cancel by user through an 'x' button in the AC Frame
      originUrl: "https://yourwebclientapplication.com",      //partner's system web application domain, this is where is SDK loaded from
      debug: true,                                            //optional parameter, send 'true' if you want a debugging logs in the browser console
      environment: "staging"                                  //default parameter if not sent, possible values are: localhost, development, staging and production
  });

It's recommended but not mandatory that onSuccess, onDecline and onCancel functions support receiving JSON object as the Aircash Frame system on calling these functions will send a JSON object with details of the transaction. An example of the payload is below: 

  {
      "amount": 100.0,
      "currencyId": 978                                       //ISO-4217 currency code
  }

In case of user closes window by pressing 'close' button in browser, onCancel function is called with the following payload: 

  {
      "byUser": true
  }
Redirect Checkout

This integration type offers you a simple checkout solution where, with a few passed parameters, a client will be fully redirected to the Aircash Frame system and after finishing the transaction, Frame system will redirect to one of the predefined URLs depending on transaction status. Below are the required parameters for initiating a transaction: 

  {
      ... //rest of the parameters for API v2
      "successUrl": "https://domain.com/success",             //this is the URL where the client will be redirected after successfull transaction
      "declineUrl":"https://domain.com/decline",              //this is the URL where the client will be redirected after declined transaction
      "cancelUrl":"https://domain.com/cancel",                //this is the URL where the client will be redirected after manual transaction cancelation
      "originUrl": null,                                      //this must be 'null' when using this type of checkout
      ... //rest of the parameters for API v2
  }

After a successful transaction initiate Aircash Frame will return the following response: 

  {
      "transactionId": "c125ba7d-afb5-4ce5-ac3c-33af420200cf",
      "url": "https://frame.aircash.eu/c125ba7d-afb5-4ce5-ac3c-33af420200cf"
  }

When the partner's system gets a response from Aircash Frame it's necessary to call the SDK function which can be done as in the following example: 

  new AircashFrame.RedirectCheckout({
      transactionId: response.transactionId,                  //this is a "TransactionId" parameter from the Aircash Frame system response
      debug: true,                                            //optional parameter, send 'true' if you want a debugging logs in the browser console
      environment: "staging"                                  //default parameter if not sent, possible values are: localhost, development, staging and production
  });

Now, the client will be fully redirected to the Aircash Frame and after finishing the transaction it will be returned (redirected) to one of the URLs defined by the partner's system on transaction initiation.

Custom implementation for redirect checkout

This is your custom implementation of Aircash Frame integration. You can display Aircash Frame on any possible way that seems the best for your requirements. For example, you can open new web browser’s window (blank or with your origin URL or any other URL) and use custom redirect to Aircash Frame URL which you will receive in the Initiate method response body. In this case you have complete control over window where Aircash Frame is displayed, you can implement your scripts to handle window open, window close and window redirect to your success URL.

FAQ


1. Q:  Is user redirected to Aircash Frame Abon and is asked to enter the 16-digit code (or scan it)?
A:  Yes, user is redirected to Aircash Frame and in the Aircash Frame user is asked to enter 16-digit Abon code.
2. Q:  Where does customer’s journey starts with Aircash Frame AcPay?
A:  Customer journey starts at the merchant's website (desktop or mobile) or native app. After customer clicks on button for example "Pay with Aircash" the Aircash Frame will pop up. Aircash Frame can be initiate either from merchant's website or native mobile app, that depends on merchant platform.
3. Q:  Where will be customer redirected after successful Aircash Frame AcPay payment?
A:  After the successful payment customer will be redirected to successURL (from the initiate request) which informs that the transaction is successfully processed.
4. Q:  How can we show Aircash Frame?
A:  You can use custom implementation to show/display Aircash Frame or you can use our Aircash Frame SDK (more about SDK examples you can find on: Aircash Simulator).
5. Q:  Are there chargebacks possible?
A:  Aircash marketplace deposit -> chargeback is possible but manually
   Aircash Frame Aircash Pay -> chargeback is possible but manually
   Aircash Frame Abon -> chargeback is not possible
6. Q:  In the withdrawal process via Aircash Frame can the Amount from Confirm payout method differ from the Initiate method?
A:  Once you initiate withdrawal via Aircash Frame and send Amount as a parameter you must reserve funds on the user's Wallet on the merchant's side. If you have manual approval of withdrawals or automatic ones once you call Confirm payout method funds will be shown on the user’s Wallet in the Aircash System. So you must send the same Amount in Confirm payout method as you sent in the Initiate method.
7. Q:  Are there returns for the withdrawals possible? (reversal of the withdrawals/payouts)
A:  Yes, returns for the withdrawals are possible manually. Customer must contact merchant and we can made returns manually if customer still has money in Aircash Wallet
8. Q:  Are you provided reconciliation files for download, either manual download or by putting them to our SFTP server or download from your SFTP server?
A:  Yes, we provide you reconciliation files for download. We can upload files on our FTP through FTPs or we can put them to your SFTP server.
9. Q:  When are you calling our NotificationURL which we define as parameter in "Initiate" method and what happens if the payment notification cannot be delivered?
A:  We are calling your NotificationURL when the transaction is processed (when a user scans the QR code and clicks in Aircash mobile app "Pay" or when the user clicks on "x" in Aircash Frame). NotificationURL is defined by your side when you are calling out the method "Initiate" to display Aircash Frame. After the transaction is processed (successful transaction, declined transaction, or canceled transaction) we are calling one time your NotificationURL. After the first call of your NotificationURL, if we receive a different response than OK(200) we will call your NotificationURL two more times.
The main task of NotificationURL is to notify your system that the transaction is completed. It is your job to check what is the status of the transaction by sending the requests to our "status" method. If, for any reason, you haven't received the notification (network issue i.e.) you still need to ensure to check the status of the transaction on our end.
For example, if we agree to have a transaction timeframe of 5 minutes (within which the user needs to confirm the payment), and you haven't received the notification, your system will need to check the transaction status without being notified. These are corner cases but still may occur.
10. Q:  What is "partnerUserId" parameter from "Initiate" method and what is it used for?
A:  partnerUserId - this value should be a unique client identifier. It can be any type of value (int, string….). We are using it for the limits calculation, and some features (like saving the phone number for future payments) which are based on the value of partnerUserId . Still, the importance of having this value depends on the type of the merchant. For example, e-commerce merchants are low-risk categorized so we could have here some random value, UUID for example (which is not fixed). If you have that info we should use it for its real purpose, otherwise, send us some random value (only because it is required).
11. Q:  What is "locale" parameter from "Initiate" method and what is it used for?
A:  locale is used to define display language of the Aircash Frame and preselect dropdown with country phone code. For example, if you want display language to be english and country phone code to be Croatia you must send locale parameter as "en-HR".

Examples in table below:

Locale Display Language Country Phone Code
"en-US" English United States of America
"en-HR" English Croatia
"en-GB" English Great Britain
"en-RO" English Romania
"hr-HR" Croatian Croatia
12. Q:  What is Aircash marketplace deposit? How does it compare to Aircash Frame implementation? Is this for mobile app native integration?
A:  Aircash marketplace deposit is a deposit method via Aircash Marketplace which is placed in the Aircash native mobile app. In the marketplace, the user will choose the merchant logo, and he will need to enter the amount and identification parameter (email or username). Compared to the Aircash Frame implementation, deposit transaction starts in the Aircash native app (not at the merchant cashier). AircashPayment integration includes the implementation of the "CheckPlayer" method and the "CreateAndConfirmPayment" method.
13. Q:  Are we expected to call Status periodically and wait for it to be moved to Pending so that the Confirmation can be called?
A:  Default behaviour of your system would be to wait for notification.  If, from any reason the notification is not received, you can call "Status" method 30 seconds after Aircash Frame timer expired to make sure there was not any network or other issues and make sure you had the right transaction status in your system (the same as in the Aircash system).
14. Q:  Is there any timer for Aircash Frame transactions? What happense once the timer expire?
A: We suggest making a timestamp of 5min for the Aircash Frame timer. Once transaction time passes (5min) and transaction timeouts, we will call your NotificationUrl and the transaction will be in status 1 – declined. If for any reason (network problems etc.) the notification is not received, you can call the Status method 30 seconds after the Aircash Frame timer expired to make sure there was not any network or other issues and make sure you had the right transaction status in your system (the same as in the Aircash system)
